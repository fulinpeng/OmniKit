2. 上一个订单的止盈和本次订单的开仓恰好都在同一个地方同一个时间点上，导致下单后，设置的止盈止损被上一个订单的平仓逻辑给重置了

用 Pine 的执行顺序解释一下：

在 当前这根 bar 里脚本执行时，strategy.position_size 代表的是 这一根 bar 开始时的仓位（上一根 bar 收盘后的状态）；

也就是说：

上一根 bar 触发了 Short Exit；

这一根 bar 开始时 strategy.position_size == 0，strategy.position_size[1] < 0；

这一根 bar 里脚本执行顺序是：

先计算信号 & 下新的空单（Short entry）并计算 shortStopLoss/shortTakeProfit；

然后才执行我们写的“平仓后重置变量”逻辑：



3. 你在 entry 时已经根据 SSL1Down 算好了 longStopLoss 和 longTakeProfit，这里又用 avgPrice * 0.999 / 1.001 去强行校正，有时会把原本合理的止损/止盈改得非常靠近入场价，甚至在某些 Backtest 设置下这一条 exit 单可能在下一根 bar 被判定为“无效价位”，直接忽略。


4. 代码逻辑：
    止损止盈只在开仓时计算一次

    longStopLoss    := ssl1Down
    longInitialStopLoss := longStopLoss
    longTakeProfit  := longEntryPrice + (longEntryPrice - longStopLoss) * riskRewardRatio


    并且在这里就保证：

    longStopLoss < longEntryPrice

    longTakeProfit > longEntryPrice
    如果算出来不符合，就做最小修正。

    不再在后面用 0.999 / 1.001 去改单价
    之前这一段：

    if stopPrice >= avgPrice
        stopPrice := avgPrice * 0.999
    if tpPrice <= avgPrice
        tpPrice := avgPrice * 1.001


    我直接删掉了。
    现在 strategy.exit 里用的就是白框那两个值本身。

    移动止损逻辑依然保留，但完全受 enableTrailingStop 控制

    enableTrailingStop=false 时：不改 longStopLoss，不改 longTakeProfit，strategy.exit 始终用初始值。
